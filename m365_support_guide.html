01
NDR 550 5.7.1 — Sender Not Authorized / Unable to Relay
SuddenSupportHigh Severity
▼
Symptoms
  • Users or apps receive 550 5.7.1 NDR when sending email externally
  • Message trace shows "Sender identity does not match accepted domain"
  • Works in OWA but fails from desktop Outlook or scanner/application relay
  • Shared mailbox can receive but cannot send
Root Cause
Most common: An application, printer, or shared mailbox is trying to relay via EXO SMTP but is not correctly configured — wrong connector, no Send As permission, or SMTP AUTH disabled. In hybrid, the outbound connector may be routing mail back to on-premises instead of directly out.
Resolution Steps
  1. Run Message Trace (Admin Center → Exchange → Mail flow → Message trace) to capture the exact rejection reason and which connector was used.
  2. Check Accepted Domains: EAC → Mail flow → Accepted domains — sender domain must be listed (Authoritative or Internal Relay).
  3. For application SMTP relay: Create a dedicated inbound connector with the app's static IP, OR enable SMTP AUTH on a service account and use port 587.
  4. For shared mailboxes: Assign "Send As" permission to the user, or enable SMTP AUTH on the shared mailbox.
  5. Check for transport rules that may be rejecting or redirecting mail: EAC → Mail flow → Rules → look for broad reject/delete actions.
  6. Verify the outbound connector is not force-routing to on-prem smart host when it should go direct.
PowerShell
# Enable SMTP AUTH on a specific mailbox Set-CASMailbox -Identity "app@contoso.com" -SmtpClientAuthenticationDisabled $false # Check accepted domains Get-AcceptedDomain | Select Name, DomainName, DomainType # Check who is on the restricted senders list Get-BlockedSenderAddress
💡 Interview Tip: Message Trace is the #1 diagnostic tool for Exchange issues. Always start there — it shows exact connector used, rejection reason, and timestamps.
02
Exchange Hybrid Cutover — Mail Stuck in On-Premises After Mailbox Migration
MigrationHigh Severity
▼
Symptoms
  • Migrated users get email delivered to old on-premises mailbox, not EXO
  • Message trace shows delivery to "Internal Relay" instead of EXO
  • External senders get NDR 550 5.1.1 for migrated users
Root Cause
MX records still point to on-premises after cutover. Or the on-prem Exchange isn't routing to EXO because the TargetDeliveryDomain (contoso.mail.onmicrosoft.com) isn't set on the remote mailbox object, or the hybrid send connector is broken.
Resolution Steps
  1. Verify MX records: Use nslookup or MXToolbox — after cutover MX must point to contoso-com.mail.protection.outlook.com.
  2. Check on-prem remote mailbox routing address: must have targetAddress = user@contoso.mail.onmicrosoft.com.
  3. Run Azure AD Connect delta sync to replicate targetAddress to Entra ID.
  4. Verify the Hybrid Configuration Wizard (HCW)-generated send connector is still active on on-prem Exchange (points to *.mail.protection.outlook.com).
  5. Check if migration batch is complete — never change MX while migration is in "Syncing" state. Only update MX after completing the batch.
  6. If EXO mailbox is SoftDeleted, restore it before re-migrating the batch.
PowerShell
# Check migration batch status Get-MigrationBatch | Select Identity, Status, SyncedCount, FailedCount # Check remote mailbox routing address (on-premises) Get-RemoteMailbox "user" | Select PrimarySmtpAddress, RemoteRoutingAddress # Find soft-deleted mailboxes in EXO Get-Mailbox -SoftDeletedMailbox | Where {$_.PrimarySmtpAddress -like "*user*"}
⚠️ Never update MX before completing migration and testing mail flow in UAT. Rushed MX changes cause exactly this scenario — lost mail for active users.
03
Outlook Keeps Prompting for Password — Modern Auth / Credential Loop
SupportSudden
▼
Symptoms
  • Outlook repeatedly prompts for credentials despite correct password being entered
  • Works in OWA (browser) but not desktop Outlook
  • Affects multiple users after a Conditional Access policy change
  • Password prompts appear every few hours even after successful sign-in
Root Cause
Top causes: (1) Modern Authentication disabled on tenant — forces Basic Auth which Microsoft has deprecated. (2) Corrupted tokens in Windows Credential Manager. (3) Conditional Access policy requiring MFA is blocking Outlook's OAuth token refresh. (4) Autodiscover returning wrong endpoint, causing profile to re-authenticate constantly.
Resolution Steps
  1. Enable Modern Authentication tenant-wide: Admin Center → Settings → Org Settings → Modern Authentication → Enable.
  2. Clear Credential Manager: Control Panel → Credential Manager → Windows Credentials → Remove all "MicrosoftOffice" entries.
  3. Delete and recreate the Outlook profile: Control Panel → Mail → Profiles → Remove old profile, let Autodiscover build a new one.
  4. Test Autodiscover: CTRL + right-click Outlook taskbar icon → Test Email AutoConfiguration — HTTPS Autodiscover must succeed (not SRV or HTTP fallback).
  5. If CA-related: Entra → Security → Conditional Access — check if "Require MFA" policy is causing token expiry. Review token lifetime policies.
  6. Check if Basic Auth is fully blocked via an Authentication Policy — clients using Basic Auth will loop infinitely if it's blocked.
PowerShell
# Check if Modern Auth is enabled Get-OrganizationConfig | Select OAuth2ClientProfileEnabled # Enable Modern Auth Set-OrganizationConfig -OAuth2ClientProfileEnabled $true # Check authentication policies Get-AuthenticationPolicy | Select Name, AllowBasicAuthOutlookService
04
SPF / DKIM / DMARC Failures — Legitimate Email Marked as Spam
SupportProject
▼
Symptoms
  • Outbound emails land in recipients' junk/spam folders
  • Email headers show spf=fail or dmarc=fail
  • Started after migrating from on-premises to EXO (IPs changed)
  • External partners report emails being rejected by their mail gateway
Root Cause
SPF record doesn't include Microsoft's IP range (spf.protection.outlook.com). DKIM signing not enabled in EXO. DMARC set to p=reject before verifying SPF/DKIM alignment. During migration, old on-premises IPs removed from SPF too early, or too many DNS lookups causing SPF permerror (10-lookup limit).
Resolution Steps
  1. Fix SPF: DNS TXT record must include: v=spf1 include:spf.protection.outlook.com -all (remove old IPs post-migration).
  2. Enable DKIM in EXO: Defender Portal → Email & Collaboration → Policies → DKIM → Enable for your domain, then add the two CNAME records to DNS.
  3. Configure DMARC: Start with p=none for monitoring: v=DMARC1; p=none; rua=mailto:dmarc@yourdomain.com
  4. Analyze DMARC reports (use dmarcanalyzer.com or valimail.com) to find unauthorized senders.
  5. Gradually move from p=none → p=quarantine → p=reject only after confirming all legitimate mail passes SPF/DKIM.
  6. Validate with MXToolbox.com — check SPF, DKIM, and DMARC from external DNS perspective.
💡 Key principle: Never set DMARC p=reject on day 1. Always start with p=none, analyze for 2–4 weeks, fix legitimate senders, then tighten policy.
05
Distribution List Not Receiving External Email — RequireSenderAuthenticationEnabled
SupportSudden
▼
Symptoms
  • External senders get NDR when emailing the distribution list
  • Internal users can email the DL without issue
  • Message trace shows "Recipient not found" or 550 5.4.1
  • DL appears correctly in GAL
Root Cause
The DL has RequireSenderAuthenticationEnabled = True (default for DLs synced from on-premises AD). This only allows authenticated (internal) senders. External senders are unauthenticated so they are rejected. This setting must be changed in on-premises AD if the DL is synced — changes in EXO will be overwritten by the next AAD Connect sync.
Resolution Steps
  1. Verify the setting: Get-DistributionGroup "DLName" | Select RequireSenderAuthenticationEnabled — if True, that's the cause.
  2. If DL is cloud-only: Set-DistributionGroup "DLName" -RequireSenderAuthenticationEnabled $false
  3. If DL is synced from on-premises: Make the change on-prem AD using Set-DistributionGroup (on-prem Exchange) or edit msExchRequireAuthToSendTo AD attribute, then wait for AAD Connect sync.
  4. Consider alternatives for security: Instead of allowing all external senders, add specific allowed external senders using -AcceptMessagesOnlyFrom or set up a moderated DL.
PowerShell
# Fix DL to accept external email (cloud-only DL) Set-DistributionGroup -Identity "dl-support@contoso.com" -RequireSenderAuthenticationEnabled $false # Verify the change Get-DistributionGroup "dl-support@contoso.com" | Select RequireSenderAuthenticationEnabled
06
Hybrid Free/Busy Not Working — Calendar Scheduling Shows "No Information"
MigrationProject
▼
Symptoms
  • On-prem users see "No Information" for EXO users in Scheduling Assistant
  • EXO users cannot see free/busy of on-prem users
  • Error: "Calendar data can't be retrieved" in Outlook
  • OWA shows free/busy but Outlook desktop does not
Root Cause
Free/busy in hybrid uses OAuth between on-premises Exchange and EXO via the Organization Relationship. If the OAuth trust is broken, the Availability Address Space is wrong, or the on-premises EWS (Exchange Web Services) URL isn't published correctly, free/busy fails. Running the HCW fixes this automatically but the config can drift post-deployment.
Resolution Steps
  1. Test OAuth connectivity from on-premises Exchange: Test-OAuthConnectivity -Service EWS -TargetUri https://outlook.office365.com/ews/exchange.asmx -Mailbox admin@contoso.com -Verbose
  2. Check Organization Relationship in EXO: Get-OrganizationRelationship — verify FreeBusyAccessEnabled = True, TargetApplicationUri, and TargetAutodiscoverEpr are correct.
  3. Re-run the Hybrid Configuration Wizard (HCW) — this is the fastest fix and reconfigures OAuth, connectors, and organization relationships automatically.
  4. Verify on-prem Autodiscover is externally resolvable — EXO needs to discover the on-prem EWS endpoint.
  5. Check on-prem Exchange certificate has all required SANs (mail, autodiscover, EWS).
PowerShell
# Test OAuth (run on on-premises Exchange server) Test-OAuthConnectivity -Service EWS -TargetUri https://outlook.office365.com/ews/exchange.asmx -Mailbox "admin@contoso.com" -Verbose # Check EXO Organization Relationship Get-OrganizationRelationship | Select Name, DomainNames, FreeBusyAccessEnabled, TargetApplicationUri
07
Mailbox Migration Stuck at 95% or Failing with BadItemLimit Error
MigrationHigh Severity
▼
Symptoms
  • Migration batch stuck at 95% for hours — never completing
  • Specific mailboxes fail with "MigrationPermanentException" or "CorruptedItemDetected"
  • Migration report shows BadItemCount exceeding configured threshold
  • Large mailboxes (50GB+) always fail or time out
Root Cause
Corrupted items in source mailbox (common in Exchange 2010/2013). The migration batch hit its BadItemLimit threshold and stopped. Stuck at 95% = "final sync" phase — the mailbox is still receiving active mail and the delta sync loop won't complete until the user stops receiving messages (or you force-complete). Large Recoverable Items folder (over EXO 100GB limit) also causes failures.
Resolution Steps
  1. Get detailed failure report: Get-MigrationUserStatistics -Identity "user@contoso.com" -IncludeReport | fl
  2. For BadItemLimit: Increase it on the batch (carefully — you are accepting data loss for corrupted items): Set-MigrationBatch -Identity BatchName -BadItemLimit 50 -LargeItemLimit 10
  3. For corrupted items: Run NewMailboxRepairRequest on source mailbox before retrying migration.
  4. For stuck at 95%: Schedule Complete-MigrationBatch during off-peak hours when the mailbox is least active. This forces final cutover.
  5. For large Recoverable Items: Remove unnecessary holds temporarily or use New-ComplianceSearch to reduce item count before migrating.
PowerShell
# Get per-user migration failure details Get-MigrationUserStatistics -Identity "user@contoso.com" -IncludeReport | fl # Repair source mailbox corruption (run on-premises) New-MailboxRepairRequest -Mailbox "user@contoso.com" -CorruptionType ProvisionedFolder,SearchFolder,AggregateCounts # Force complete the migration batch Complete-MigrationBatch -Identity "BatchName"
08
Shared Mailbox Not Receiving Email — ProxyAddress Accidentally Removed
SupportSudden
▼
Symptoms
  • Emails to shared mailbox address bounce with 550 5.4.1 or "Recipient not found"
  • Mailbox is visible and working for internal users but external delivery fails
  • Message trace: recipient lookup returns no result for the SMTP address
Root Cause
The shared mailbox's primary SMTP address was accidentally removed — either by an admin, a PowerShell script, or an AAD Connect sync conflict that wrote back a different set of ProxyAddresses. The mailbox exists but its email address is gone, so inbound mail cannot be delivered.
Resolution Steps
  1. Verify current email addresses: Get-Mailbox "SharedMailbox" | Select -ExpandProperty EmailAddresses
  2. Re-add the missing primary address (uppercase SMTP = primary): Set-Mailbox "SharedMailbox" -EmailAddresses @{Add="SMTP:helpdesk@contoso.com"}
  3. If cloud-synced from on-prem: Fix the ProxyAddresses attribute in on-premises AD and wait for AAD Connect delta sync — don't fix in EXO as it gets overwritten.
  4. Wait 15–30 minutes for mail routing to update, then test delivery with a fresh message trace.
PowerShell
# View all current email addresses Get-Mailbox "SharedMailboxName" | Select -ExpandProperty EmailAddresses # Re-add the missing primary SMTP address Set-Mailbox "SharedMailboxName" -EmailAddresses @{Add="SMTP:helpdesk@contoso.com"}

Litigation Hold Not Protecting Items — Items Still Being Purged
ProjectHigh Severity
▼
Symptoms
  • Litigation Hold was applied but items are still disappearing after the deleted item retention period
  • eDiscovery search returns fewer items than expected for a user on hold
  • Compliance team reports gaps in preserved email records
Root Cause
Litigation Hold was set but LitigationHoldDuration was not set to Unlimited — items are held for a finite period then eligible for purge. Or the hold was set AFTER the items were already purged. Managed Folder Assistant (MFA) runs every 7 days and processes deletions — if hold isn't properly applied, MFA cleans up Recoverable Items after the default 14-day retention.
Resolution Steps
  1. Verify hold status and duration: Get-Mailbox "user" | Select LitigationHoldEnabled, LitigationHoldDuration, LitigationHoldDate
  2. Set to unlimited hold: Set-Mailbox "user" -LitigationHoldEnabled $true -LitigationHoldDuration Unlimited
  3. Check Recoverable Items size — if growing, the hold is working: Get-MailboxFolderStatistics "user" -FolderScope RecoverableItems
  4. For already-purged items: Search the Recoverable Items\Purges folder using compliance search — items may still be there if within the purge window.
  5. Consider using Microsoft Purview retention policies instead of Litigation Hold for better management and reporting.
PowerShell
# Apply Litigation Hold with unlimited duration Set-Mailbox "user@contoso.com" -LitigationHoldEnabled $true -LitigationHoldDuration Unlimited # Check Recoverable Items usage Get-MailboxFolderStatistics "user@contoso.com" -FolderScope RecoverableItems | Select Name, ItemsInFolder, FolderSize
10
Transport Rules Silently Dropping or Misrouting Email
SupportSudden
▼
Symptoms
  • Emails from specific senders never arrive — no bounce, no error
  • Double disclaimers on outbound emails
  • Emails suddenly redirected to wrong mailbox after new rule was created
  • Issue started immediately after a new transport rule was added
Root Cause
A transport rule with broad conditions (e.g., applies to all senders) combined with a Delete, Redirect, or Reject action is silently consuming mail. Transport rules are evaluated in priority order — a "Stop processing more rules" action on a high-priority rule prevents lower-priority rules from firing, causing unexpected behavior.
Resolution Steps
  1. Audit all transport rules by priority: EAC → Mail flow → Rules → sort by Priority. Look for broad conditions with Delete/Redirect actions.
  2. Use Message Trace: Select affected message → "Applied rules" section shows exactly which rules fired and what action was taken.
  3. Test rules safely with Audit mode: Set-TransportRule "RuleName" -Mode AuditAndNotify before setting to Enforce.
  4. Check "Stop processing more rules" checkboxes — they short-circuit the entire rule pipeline.
  5. Disable suspected rules one-by-one to isolate the culprit, then narrow conditions before re-enabling.
PowerShell
# List all transport rules with priority order Get-TransportRule | Select Priority, Name, State, Mode | Sort Priority # Set a rule to Audit-only (safe testing) Set-TransportRule -Identity "RuleName" -Mode AuditAndNotify # Disable a problematic rule immediately Disable-TransportRule -Identity "RuleName"

Teams Meeting Audio/Video Broken — Stuck on "Connecting" (Network/Firewall)
SuddenSupportHigh Severity
▼
Symptoms
  • Users stuck on "Connecting to audio..." indefinitely during meetings
  • Video freezes or black screen — audio only also broken
  • Works perfectly on home WiFi but fails on corporate network
  • Entire office affected simultaneously — single user issues are different
Root Cause
Teams uses UDP ports 3478–3481 and 50000–59999 for real-time media. Corporate firewalls blocking UDP cause Teams to attempt TCP fallback or relay servers, resulting in poor quality or failure. Running Teams media through a proxy (instead of bypassing it) adds massive latency. Microsoft classifies Teams traffic into Optimize/Allow/Default categories — Optimize traffic MUST bypass proxy.
Resolution Steps
  1. Run Teams Network Assessment Tool on affected machines — measures packet loss, jitter, and round-trip time to Microsoft media relays.
  2. Check proxy bypass: Add Microsoft's Optimize-category IPs and FQDNs (aka.ms/o365ips) to proxy bypass/split-tunnel list. Teams media must NOT go through a proxy.
  3. Open required firewall ports: UDP 3478–3481 outbound, UDP/TCP 50000–59999 outbound to *.teams.microsoft.com and *.lync.com.
  4. Configure QoS DSCP markings: Audio = EF (46), Video = AF41 (34), Sharing = AF21 (18) to prioritize Teams traffic at network layer.
  5. Use Call Quality Dashboard (CQD) in Teams Admin Center to identify poor streams by building, subnet, or device type.
  6. Check Teams Admin Center → Users → Select user → Call history → View per-call metrics (jitter, packet loss, round-trip).
💡 Use Call Analytics for individual call troubleshooting and CQD for organization-wide quality trends. Both are in Teams Admin Center and are essential diagnostic tools.

Guest Access Not Working — "You Don't Have Access to This Team"
ProjectSupport
▼
Symptoms
  • Guests receive invitation but get "Access denied" when trying to join
  • "Your IT admin has restricted access to this feature" error
  • Guest can see team listed but all channels are empty or inaccessible
  • Guest access worked before but stopped after a policy change
Root Cause
Guest access requires 3 independent controls ALL set correctly: (1) Entra ID External Collaboration settings, (2) Teams Admin Center guest access toggle, (3) individual team-level guest settings. Any one missing = blocked. Also: Conditional Access policies targeting "All users" apply to guests too. SharePoint external sharing controls Teams file access.
Resolution Steps
  1. Entra ID: Azure Portal → Entra ID → External Identities → External collaboration settings → Allow guest invitations.
  2. Teams Admin Center: Org-wide settings → Guest access → Turn ON. Enable calling, meeting, and messaging for guests.
  3. Per-team: Team Settings → Guest permissions → check appropriate permissions are granted.
  4. SharePoint Admin Center: Policies → Sharing → Allow sharing with new and existing guests (minimum).
  5. Conditional Access: Check if any CA policy forces compliant device or MFA requirement that guest devices can't satisfy — add guest exclusion if appropriate.
  6. Note: Guest access changes can take up to 24 hours to propagate through the system.
PowerShell
# Check guest access settings Get-CsTeamsClientConfiguration | Select AllowGuestUser # Check federation configuration Get-CsTenantFederationConfiguration | Select AllowTeamsConsumer, AllowFederatedUsers, AllowPublicUsers
03
Teams Direct Routing — PSTN Calls Failing (SIP 403/503) or One-Way Audio
ProjectSuddenHigh Severity
▼
Symptoms
  • External PSTN calls fail with SIP 403 Forbidden or 503 Service Unavailable
  • Calls connect but only one side can hear audio (one-way audio)
  • All calls broke after SBC TLS certificate was renewed
  • Specific users can't make external calls — others on same SBC can
Root Cause
SIP 403 = User not licensed for Phone System OR no voice routing policy assigned. One-way audio = Media bypass misconfigured causing NAT traversal failure, or SRTP key mismatch between Teams and SBC. After certificate renewal, the SBC FQDN on the cert must exactly match what's registered in Teams Admin Center. TLS 1.2 is minimum — TLS 1.0/1.1 causes silent failures.
Resolution Steps
  1. Check SBC health: Teams Admin Center → Voice → Direct Routing → View SBC status, certificate validity, and TLS connection state.
  2. Verify user licensing: User needs Teams Phone (Phone System) license AND a PSTN connectivity option (Calling Plan or Direct Routing). Check in Admin Center → Users → Licenses.
  3. Verify number assignment: Get-CsOnlineUser -Identity "user" | Select LineUri, EnterpriseVoiceEnabled
  4. For one-way audio: Temporarily disable media bypass to test: Set-CsOnlinePSTNGateway -Identity sbc.contoso.com -MediaBypass $false. If this fixes it, reconfigure media bypass properly.
  5. Post certificate renewal: Re-bind TLS cert on SBC, verify FQDN matches exactly what's in Teams (including case), and test TLS handshake.
  6. Review voice routing policy: Ensure online voice routing policy is assigned and routing rules match the called number format.
PowerShell
# Check Direct Routing SBC configuration Get-CsOnlinePSTNGateway | Select Identity, Fqdn, SipSignalingPort, Enabled, MediaBypass # Assign phone number to user Set-CsPhoneNumberAssignment -Identity "user@contoso.com" -PhoneNumber "+14255550100" -PhoneNumberType DirectRouting # Assign voice routing policy Grant-CsOnlineVoiceRoutingPolicy -Identity "user@contoso.com" -PolicyName "International"
04
Teams Upgrade from Skype for Business — Chats Routing to Wrong App
MigrationProject
▼
Symptoms
  • After "migrating to Teams," some users receive chats in Skype for Business instead of Teams
  • Meeting invites still show Skype join links, not Teams links
  • External federated users reach users on Skype instead of Teams
  • Islands mode causes split conversations — some in Teams, some in SfB
Root Cause
The Teams coexistence mode is set to "Islands" — both SfB and Teams run simultaneously and external routing is unpredictable. Users must be moved to "TeamsOnly" mode to route all communication through Teams. Existing Skype meeting links won't automatically convert — the Meeting Migration Service (MMS) must run to update calendar invites.
Resolution Steps
  1. Move users to TeamsOnly: Grant-CsTeamsUpgradePolicy -Identity user@contoso.com -PolicyName UpgradeToTeams
  2. Run Meeting Migration Service to convert all existing Skype meeting links to Teams: Start-CsExMeetingMigration -Identity user@contoso.com -TargetMeetingType Teams
  3. For tenant-wide upgrade: Grant-CsTeamsUpgradePolicy -Global -PolicyName UpgradeToTeams
  4. For on-premises SfB hybrid: Use Move-CsUser to move users to the cloud before granting TeamsOnly mode.
  5. Communicate to users: In TeamsOnly mode, SfB client is read-only for history. All new chats/calls come to Teams.
PowerShell
# Check current upgrade mode for a user Get-CsOnlineUser "user@contoso.com" | Select TeamsUpgradeEffectiveMode # Upgrade user to Teams Only Grant-CsTeamsUpgradePolicy -Identity "user@contoso.com" -PolicyName UpgradeToTeams # Trigger meeting migration and check status Start-CsExMeetingMigration -Identity "user@contoso.com" -TargetMeetingType Teams Get-CsMeetingMigrationStatus -Identity "user@contoso.com"
05
Teams Policies Not Applying — Feature Available When It Should Be Blocked
ProjectSupport
▼
Symptoms
  • User can record meetings even though recording policy says disabled
  • Policy change deployed but user experience unchanged after 24+ hours
  • Group-based policy assignment seems to have no effect
Root Cause
Teams policy precedence: Direct user assignment > Group-based assignment > Global (org-wide default). If a user has a DIRECT policy assignment, group policies are completely ignored for that user. Multiple groups assigning different policies are ranked — highest-rank (lowest number) wins. Policy propagation can take up to 4 hours.
Resolution Steps
  1. Check effective policy: Teams Admin Center → Users → Select user → Policies tab — this shows what's actually applied vs. what's assigned.
  2. Remove incorrect direct assignment: Grant-CsTeamsMeetingPolicy -Identity "user" -PolicyName $null (falls back to group or global).
  3. For group policy conflicts: Teams Admin Center → Group policy assignments → review all groups and their rank numbers.
  4. Use Get-CsUserPolicyAssignment to audit all policy assignments for any user across all policy types.
PowerShell
# Check all policy assignments for a user Get-CsUserPolicyAssignment -Identity "user@contoso.com" # Remove direct meeting policy (fall back to group/global) Grant-CsTeamsMeetingPolicy -Identity "user@contoso.com" -PolicyName $null # Check group-based assignments Get-CsGroupPolicyAssignment -PolicyType TeamsMeetingPolicy
06
Teams Files Tab Blank — SharePoint Permission Broken by Admin
SuddenSupport
▼
Root Cause
Teams channels are backed by SharePoint document libraries. If a SharePoint admin directly modifies permissions on the backing site — breaking inheritance or removing the Office 365 Group — Teams members lose file access. This is extremely common when SharePoint and Teams admins work in silos.
Resolution Steps
  1. Navigate to the SharePoint site: Teams → Files tab → Open in SharePoint.
  2. Check site permissions: Settings → Site Permissions → Verify the Office 365 Group (same name as the Team) is listed with correct permissions (Members = Edit, Owners = Full Control).
  3. If inheritance was broken: Site Settings → Site Permissions → check document library → restore inheritance or re-add the O365 group.
  4. Rule: Never directly remove the O365 group from a Teams-connected SharePoint site. Manage membership through Teams or the M365 Group.
07
Teams Desktop App — Blank White Screen / Won't Load
SuddenSupport
▼
Symptoms
  • Teams opens but shows only a white/blank screen
  • Splash screen appears then freezes permanently
  • Affects multiple users after Windows Update or Teams update
  • Teams.microsoft.com in browser works fine
Root Cause
Teams app cache corruption — very common after Windows updates or abrupt shutdowns. The Electron-based Teams app (classic) or WebView2-based new Teams stores cache that becomes inconsistent. For new Teams (2023+): missing or outdated Edge WebView2 Runtime causes launch failure.
Resolution Steps
  1. Clear classic Teams cache: Close Teams → Navigate to %appdata%\Microsoft\Teams → Delete all subfolders EXCEPT: downloads and meeting-addin → Restart Teams.
  2. For new Teams cache: %localappdata%\Packages\MSTeams_8wekyb3d8bbwe\LocalCache — clear contents.
  3. Install/update Edge WebView2 Runtime (required for new Teams) — download from Microsoft.
  4. Repair Teams: Settings → Apps → Microsoft Teams → Advanced Options → Repair. If that fails, use Reset.
  5. Re-sign in: File → Sign Out → Clear credentials from Windows Credential Manager → Sign back in.
  6. If GPO related: Check AppLocker/Software Restriction Policies — Teams executables may be blocked after Windows Update changed paths.
💡 Classic vs New Teams: Cache paths are completely different. Always confirm which version the user has before clearing cache. "New Teams" uses a fundamentally different rendering engine.
08
Teams Meeting Recording Not Appearing or Saved to Wrong Location
SupportProject

Root Cause

  4. Concurrency control: Flow settings → Concurrency control → Limit to 1 concurrent run to prevent parallel race conditions.
