EOP + Proofpoint & Mimecast Integration
How third-party email security gateways work with Exchange Online Protection — inbound and outbound
1
What is EOP?

Exchange Online Protection (EOP) is Microsoft's built-in cloud-based email filtering service that is automatically included with every Exchange Online mailbox. It performs spam filtering, malware scanning, and anti-phishing analysis on all inbound and outbound mail. When you subscribe to Defender for Office 365, EOP remains the first layer — Defender adds advanced threat protection on top of EOP.

🏗️ EOP vs Defender for Office 365
EOP (Included in all M365 plans): Anti-spam, anti-malware, connection filtering, spoof intelligence, outbound spam limits
Defender for Office 365 Plan 1 (E3 add-on): Safe Attachments (sandbox detonation), Safe Links (real-time URL check), anti-phishing with mailbox intelligence
Defender for Office 365 Plan 2 (E5 included): All P1 + Attack Simulator, Threat Explorer, AIR (Automated Investigation & Response), advanced hunting
2
Why Organizations Use Proofpoint or Mimecast WITH EOP

Many enterprises had Proofpoint or Mimecast before they moved to M365. Instead of replacing them, these tools are configured as a front-end security gateway that processes all email before it reaches EXO/EOP. Common reasons: legacy compliance archiving in Mimecast, superior phishing detection in Proofpoint, continuity service during outages, and email DLP features not yet migrated to Purview.

MX Record Setup: The domain's MX record points to Proofpoint/Mimecast, NOT to EXO. The world's mail servers deliver to the third-party gateway first. After filtering, Proofpoint/Mimecast forwards to EXO via a dedicated inbound connector that only accepts mail from the gateway's IP ranges.

INBOUND CONFIG  Critical: SCL=-1 on inbound connector
When Proofpoint/Mimecast delivers to EXO, you must configure the inbound connector to bypass EOP spam filtering (set SCL=-1). This prevents EOP from re-filtering mail that the gateway already approved, which would cause double-filtering and false rejections. The connector is locked to the gateway's published IP ranges — any mail NOT from those IPs is rejected, preventing attackers from bypassing the gateway by delivering directly to EXO.
💡 How to verify gateway is the only inbound path: Check the inbound connector "Restrict by IP" setting, and verify no mail is being accepted directly to *.protection.outlook.com without first going through the gateway. Run a message trace to confirm all inbound mail shows the gateway IP in the first "Received" header hop.

For outbound, an EXO outbound connector is configured to route all outgoing mail through the gateway's SMTP relay servers. This allows Proofpoint/Mimecast to apply outbound DLP, email encryption, disclaimers, and DKIM signing before delivery to external recipients.

⚠️ DKIM signing consideration: If the gateway re-writes or adds headers, it may break EXO-signed DKIM. Either configure DKIM signing on the gateway itself (using the domain's selector), OR configure the gateway to not modify headers that affect the DKIM signature.

Key EOP outbound checks: Outbound spam score (messages per minute limits), DKIM signing (uses your domain's CNAME-based selector configured in EXO), SPF check (EXO sends from *.protection.outlook.com which is in spf.protection.outlook.com), and Outbound connector routing (smart host or direct send).

✅ Common Issue in This Scenario
User reports external recipient says their email went to spam. Root cause 90% of time: DKIM not enabled in EXO, or DMARC set to p=none with failing SPF. Fix: Enable DKIM in EXO for the domain + Verify SPF includes spf.protection.outlook.com + Gradually move DMARC to p=quarantine.

EOP inbound layers (applied in order): 1) Connection filtering — checks sender IP against Microsoft's block lists and your custom IP Allow/Block list. 2) Anti-malware — scans attachment content. 3) Mail flow rules (transport rules) — any admin-configured rules fire here. 4) Anti-spam — assigns SCL score. 5) Anti-phishing — spoof intelligence, impersonation checks. 6) Safe Attachments/Safe Links (if Defender P1/P2 licensed).

✅ Common Issue in This Scenario
Legitimate vendor emails going to Junk. Check order: Run Message Trace → Look at SCL value and Applied Rules. If SCL=5+, the sender's domain/IP has reputation issues. Fix: Add sender to Safe Senders list in anti-spam policy (not Outlook client — do it at tenant level), or whitelist specific sender IP in Connection Filter "IP Allow List" in EOP.

Internal-to-internal mail stays within EXO and does NOT go through the full EOP anti-spam stack (because senders are already authenticated M365 users). Hybrid Architecture — What "Managed from On-Premises" Means

In a classic Exchange Hybrid setup, user mailboxes live in Exchange Online (cloud) but key management tasks remain on-premises. This is because the organization's source of truth for user accounts is on-premises Active Directory — synchronized to Entra ID via AAD Connect.

Online Archiving in Hybrid

Exchange Online Archiving (EOA) provides a cloud-based archive mailbox that can be paired with either an on-prem mailbox OR a cloud mailbox. In hybrid, a common configuration is: primary mailbox on-premises, archive mailbox in Exchange Online. This reduces on-prem storage while keeping the primary mailbox locally for performance.

In-Place Archive
Secondary mailbox attached to a user's primary mailbox. Accessible in Outlook under "Online Archive — Name." Default 50GB (expandable to 1.5TB with auto-expanding archive on E3/E5).
Retention Policy (MRM)
Messaging Records Management — moves old items to archive automatically. Default policy moves items 2+ years old to Archive folder.
Archive Quota Warning
Warning at 45GB (default), hard limit at 50GB for EOA Plan 1. E3/E5 enables auto-expanding archive — no practical limit.
AutoExpanding Archive
Enabled per-mailbox or tenant-wide. When archive reaches 90GB, Microsoft provisions additional storage automatically. Max ~1.5TB total.

SCL and BCL Values — Spam and Bulk Confidence Levels

SCL (Spam Confidence Level) is a number assigned to each inbound message by EOP's spam filtering engine. It ranges from -1 to 9. Higher = more likely spam. Admins configure at what SCL threshold messages are moved to Junk or quarantined.

SCL ValueMeaningDefault Action
SCL = -1Bypass spam filtering completely. Used for trusted inbound connectors (e.g., Proofpoint delivery).Delivered directly to inbox — skip all spam checks
SCL = 0–1Not spam. Clean message.Delivered to inbox
SCL = 5–6Spam detected (medium confidence)Moved to Junk Email folder
SCL = 7–8High confidence spamMoved to Junk Email folder
SCL = 9Very high confidence spamQuarantined (admin must release)

BCL (Bulk Complaint Level) measures whether a message is bulk/commercial email (like newsletters). Scale 0–9. Higher = more complaints about this sender. BCL≥7 is typically treated as spam by default EOP policies.

✅ How to Use SCL for Troubleshooting
User reports legitimate email going to Junk. Pull the message headers. Find X-MS-Exchange-Organization-SCL: 5 (or higher). This is the SCL assigned. Also find X-Forefront-Antispam-Report header — it lists all spam filter checks and their results. Share with anti-spam admin to whitelist the sender or tune the policy.
3
Message Header Analyzer — How to Use It

The Microsoft Message Header Analyzer is a free web tool at aka.ms/mha (or available in Outlook → File → Properties to get headers). It decodes the raw email headers into human-readable format showing every server hop, delay, and filtering decision.

📋 Step-by-Step: Analyzing a Problematic Email
Step 1: Get the full message headers. In Outlook: Open email → File → Properties → Internet Headers box → Select all → Copy.
Step 2: Go to aka.ms/mha → Paste headers → Click "Analyze headers"
Step 3: Check "Received" section — shows every server the email passed through with timestamps. Look for unusual hops or long delays (indicates server queue issues).
Step 4: Check Authentication Results — shows SPF (pass/fail/softfail), DKIM (pass/fail), and DMARC (pass/fail) results.
Step 5: Check X-Forefront-Antispam-Report — contains: SCL value, BCL value, SFV (spam filter verdict), IPV (IP verdict), and which specific rule fired.
Step 6: Check X-MS-Exchange-Organization-SCL — the final SCL assigned before delivery.
Header FieldWhat to Look ForWhat It Tells You
Authentication-Resultsspf=pass/fail, dkim=pass/fail, dmarc=pass/failWhether sender's domain authentication is set up correctly

SharePoint Point-in-Time Restore

SharePoint Online supports site-level Point-in-Time Restore — restore an entire site collection to any point in the past 30 days. This is different from file version history (which restores individual files). PIT restore rolls back the entire site — all files, lists, pages, and settings — to a previous state.

🔄 How to Perform PIT Restore
Via Admin Portal: SPO Admin Center → Active Sites → Select site → Restore (clock icon) → Choose date/time → Confirm
Via PowerShell: Restore-SPODeletedSite for deleted sites, or use SharePoint's built-in restore from the site settings
Warning: PIT restore is destructive for content added AFTER the restore point. All changes since the selected restore point are overwritten. Always warn users before restoring.
⚠️ Limitations: PIT restore only available within 30 days. Changes made by the restore operation cannot be undone. Teams-connected sites may lose Teams channel content if restored without coordinating the Teams data. Always create an export backup before performing PIT restore in production.

3
The Preservation Principle — Conflict Resolution

When multiple retention policies or labels apply to the same content with conflicting actions, Microsoft uses the Preservation Principle:

📋 Priority Rules (in order)
1. Retain ALWAYS wins over Delete. If one policy says retain for 5 years and another says delete after 1 year — the item is retained for 5 years, THEN deleted.
2. Longer retention wins over shorter. Two retain policies: 5 years vs 7 years → retained for 7 years.
3. Explicit delete later wins over implicit. If both policies only have delete actions, the longer one wins.
4. Labels win over policies for individual items. A label applied to a specific document overrides a blanket retention policy for that document.

Common L1/L2 PowerShell Scripts — Exchange Online
# CONNECT to Exchange Online Connect-ExchangeOnline -UserPrincipalName admin@contoso.com # Check mailbox size and quota status Get-MailboxStatistics "user@contoso.com" | Select DisplayName, TotalItemSize, ItemCount, DatabaseName # Run message trace (last 48 hours for specific sender) Get-MessageTrace -SenderAddress sender@external.com -StartDate (Get-Date).AddDays(-2) -EndDate (Get-Date) | Select Received, SenderAddress, RecipientAddress, Subject, Status # Check mailbox permissions (Full Access) Get-MailboxPermission -Identity "shared@contoso.com" | Where {$_.User -ne "NT AUTHORITY\SELF"} # Check if Litigation Hold is enabled Get-Mailbox "user@contoso.com" | Select LitigationHoldEnabled, LitigationHoldDate, LitigationHoldDuration # Enable Litigation Hold Set-Mailbox "user@contoso.com" -LitigationHoldEnabled $true -LitigationHoldDuration Unlimited # Check all forwarding configured on a mailbox Get-Mailbox "user@contoso.com" | Select ForwardingAddress, ForwardingSmtpAddress, DeliverToMailboxAndForward # Get all mailboxes approaching quota (over 45GB) Get-Mailbox -ResultSize Unlimited | Get-MailboxStatistics | Where {$_.TotalItemSize.Value.ToGB() -gt 45} | Select DisplayName, TotalItemSize

Common L1/L2 Scripts — Teams, SharePoint, Entra
### TEAMS PowerShell ### Connect-MicrosoftTeams # Check user's Teams upgrade mode Get-CsOnlineUser "user@contoso.com" | Select TeamsUpgradeEffectiveMode, OnPremLineURI # Check Teams policies assigned to user Get-CsUserPolicyAssignment -Identity "user@contoso.com" # Check Direct Routing SBC health Get-CsOnlinePSTNGateway | Select Identity, Enabled, SipSignalingPort, FailoverTimeSeconds ### SHAREPOINT PowerShell ### Connect-SPOService -Url https://contoso-admin.sharepoint.com # Get all sites with storage usage Get-SPOSite -Limit All | Select Url, StorageUsageCurrent, StorageQuota | Sort StorageUsageCurrent -Desc # Set sharing level on a site Set-SPOSite -Identity https://contoso.sharepoint.com/sites/HR -SharingCapability ExternalUserSharingOnly # Restore a deleted site Restore-SPODeletedSite -Identity https://contoso.sharepoint.com/sites/OldProject ### ENTRA ID / GRAPH PowerShell ### Connect-MgGraph -Scopes "User.Read.All", "Directory.ReadWrite.All" # Get user's last sign-in date Get-MgUser -UserId "user@contoso.com" -Property SignInActivity | Select -ExpandProperty SignInActivity # Force delta sync on AAD Connect server Start-ADSyncSyncCycle -PolicyType Delta # Run on the AAD Connect server # Revoke all user sessions (compromised account) Revoke-MgUserSignInSession -UserId "user@contoso.com"
