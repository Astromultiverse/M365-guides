1
What is Entra ID? The School Gate System

Imagine a school with 50 classrooms (50 apps like Outlook, Teams, SharePoint, Power BI). Now imagine if every student needed a different key for every classroom. That would be chaos — students losing keys, teachers issuing keys, keys getting stolen.

Instead, the school installs ONE smart badge scanning system at the main gate. You scan your badge once when you enter the school. The system recognizes you. Now every classroom door with a smart lock knows who you are and lets you in based on your permissions — without scanning again.

Entra ID (formerly Azure Active Directory) is that smart badge system for Microsoft 365. You sign in once → Entra ID verifies you → you get access to all M365 services without signing in again. This is called Single Sign-On (SSO).

🏫 The School Gate Analogy — Never Forget This
School Gate = Entra ID login page (login.microsoftonline.com)
Your School Badge = Your username + password + MFA
Badge Scanner = Entra ID authentication engine
Classroom Access List = Role-Based Access Control (RBAC)
Daily Badge Scan = Token (lasts 1 hour, refreshes silently)

When the badge scanner reads your badge, it doesn't call every classroom separately. It gives you a day pass (an access token) that every classroom recognizes all day. That's how SSO works.

🎯 Interview Question — This Gets Asked a Lot
Q: "What is the difference between authentication and authorization? Give an example."
Answer that WOWs: "Great question — these are two completely different things that happen in sequence. Authentication is like showing your ID at a hotel check-in — it proves you are who you claim to be. Authorization is what room key they give you — based on what you booked, you can access room 405 but not the executive lounge.

In M365: When I enter my username and password, Entra ID authenticates me — verifies my identity. Then it checks my assigned licenses and roles to decide what I can access — that's authorization. A regular user and a Global Admin might both successfully authenticate, but their authorization is completely different — the admin can access the Admin Center, the user cannot."

Cloud-Only vs Hybrid Identity
☁️ Cloud-Only Identity
✅User account created directly in Entra ID (M365 Admin Center)
✅Password managed entirely in Entra ID cloud
✅Simple — no sync needed
⚠️Best for small orgs or startups with no on-prem AD
📝Example: new SaaS company, no legacy IT
🏢 Hybrid Identity
✅Users created in on-premises Active Directory
✅AAD Connect syncs them to Entra ID every 30 min
✅Password reset on-prem, syncs to cloud
⚠️Most common in enterprises with existing AD infrastructure
📝Example: TCS, Wipro, banks, hospitals
🏠 Moving House Analogy for Hybrid
Imagine you have an old address book (on-prem AD) with all your contacts. You also have a new digital contacts app (Entra ID). AAD Connect is like an automatic assistant who looks at your old address book every 30 minutes and copies any new or updated contacts into your digital app — keeping both in sync. If you add a new contact in the old book, it appears in the app. But if someone updates the app directly, the old book stays the same until the next sync from the book to the app.
✅ Real World Example — Why This Matters
An engineer at Wipro creates a new employee account in Windows Server Active Directory. Within 30 minutes, AAD Connect syncs this account to Entra ID. The new employee can now sign into Microsoft 365 with the same username and password they use to log into their Windows laptop. One account, two systems, seamless experience.

Now the employee is leaving. HR disables the account in on-prem AD. Next sync: account disabled in Entra ID. The employee's access to M365, email, Teams, SharePoint — ALL gone automatically. This is why hybrid is powerful for large organizations.

Common Entra ID Issues — L1/L2 Daily Reality
  • 1
    User can't sign in after password reset on-prem: Wait for AAD Connect sync (up to 30 min) or force delta sync: Start-ADSyncSyncCycle -PolicyType Delta on the sync server. Issue: password hash not yet synced to cloud.
  • 2
    User account exists in AD but not in M365: Check if the user's OU (Organizational Unit) is in the AAD Connect sync scope. If IT created the account in an OU that's excluded from sync, it won't appear in M365.
  • 3
    "Account is locked" but user swears they didn't get the password wrong: Check both on-prem AD (Account lockout threshold) AND Entra ID Smart Lockout. They have separate lockout counters. Reset both. Check for an automated process or application using an old/wrong password.
  • 4
    Duplicate user in Entra ID after migration: Two objects (one synced from on-prem, one cloud-created) with same email. Fix: use Soft Match — AAD Connect matches on UPN or SMTP address. Or set ImmutableID on the cloud object to match the on-prem anchor. Never manually delete without verifying data impact.

Remember the sign-in flow as: WHO → PROVE → WHAT
WHO are you? (Username) → PROVE it (Password + MFA) → WHAT can you access? (Token + Roles)
Authentication = WHO + PROVE. Authorization = WHAT.

MFA — Two Locks on One Door
Types of MFA · How Each Works · When to Use Which · Real Attacks it Stops
1
Why One Password is Never Enough

Think about this: there are billions of stolen passwords for sale on the dark web right now — from previous hacks of websites you signed up for years ago. If your M365 password was stolen in a LinkedIn breach in 2016, a hacker could potentially sign into your company email TODAY.

🏧 The ATM Analogy — Perfect for Interviews
An ATM requires TWO things: your physical card (something you HAVE) and your PIN (something you KNOW). If someone steals your PIN by watching over your shoulder — they STILL can't steal your money because they don't have the card. If someone steals your card — they can't use it without the PIN.

MFA works exactly the same way. Even if your password is stolen, a hacker cannot sign in without physically having your phone. Microsoft research shows MFA blocks 99.9% of automated account compromise attacks.

✅ Real Scenario — MFA Fatigue Attack (Common in Real Companies)
A hacker in Ukraine has an employee's password (bought from a dark web breach list). They try to sign in at 2 AM. Entra ID sends an MFA push to the employee's phone. The employee is sleeping — the phone buzzes, they half-asleep tap "Approve" without reading it. The hacker is in.

This happened to Uber in 2022 — compromising their entire internal network.

Fix: Enable Number Matching in Authenticator app settings. Now the phone shows "Type 47 to approve." The hacker's sign-in page shows 47, but the employee in India sees a random notification at 2 AM with no context — they don't know what 47 to type. Attack blocked.
🎯 Interview Question
Q: "What is MFA fatigue? How would you protect against it?"
Answer: "MFA fatigue is when a hacker who has stolen a user's password repeatedly sends push notifications to the user's phone hoping they'll accidentally approve one out of frustration or by mistake — especially late at night or when their phone is buzzing repeatedly. I actually studied the Uber breach of 2022 where this exact technique was used to compromise their entire network.

To prevent it, I would enable Number Matching in Microsoft Authenticator — this requires the user to type a specific two-digit number shown on the sign-in screen into their phone app. A hacker spamming pushes at 2 AM can't provide that number to the user, so even if the user opens the app, they don't know what to type. Additionally, I'd configure Entra ID Protection to detect impossible travel sign-ins — if someone signs in from India and the same account tries to sign in from Ukraine 5 minutes later, it gets automatically blocked and flagged."

Temporary Access Pass (TAP) — The Emergency Key

Sometimes users get completely locked out — they lose their phone, change phones without migrating Authenticator, or join the company on day 1 with no way to set up MFA. TAP is the solution.

🔑 The Hotel Key Analogy
When you check into a hotel and your key card stops working at 11 PM, the front desk gives you a temporary paper slip that the door will accept just once to let you in. Tomorrow you get a proper key card. TAP works identically — it's a time-limited PIN that works ONE time (or for a short window), letting the user sign in and set up their permanent MFA without needing a phone to receive an MFA code.
# Admin creates a TAP for a locked-out user (30-min window, one-time use) New-MgUserAuthenticationTemporaryAccessPassMethod -UserId "user@contoso.com" ` -IsUsableOnce $true -LifetimeInMinutes 60 # Output will show the TAP code — share securely with user (never by email!)

What is Conditional Access? The Security Guard Who Asks Questions

Imagine a very smart security guard at a company's entrance. He doesn't just check your badge. He asks a series of questions before deciding what to do:

💂 The Smart Security Guard Analogy
Guard asks: "Who are you?" → Is this a regular employee, a guest, or an admin?
Guard asks: "Where are you coming from?" → Are you in the office building or connecting from an unknown location?
Guard asks: "What device are you using?" → Is this a company laptop or an unknown personal device?
Guard asks: "What do you want to access?" → The cafeteria (low risk) or the server room (high risk)?
Guard asks: "Does this seem risky?" → Is your account flagged for suspicious activity?

Based on ALL these answers, the guard decides: Let in freely, Let in but require extra verification (MFA), or Block completely. Conditional Access does EXACTLY this — automatically, for every sign-in, in milliseconds.

Report-Only Mode — The Most Critical Safety Feature

The biggest mistake admins make: enabling a new CA policy directly in production without testing it first. This can instantly lock everyone out of M365 — a P1 incident.

🎮 The Game Difficulty Analogy
Imagine you're about to change a game's difficulty to "Extreme" for all 1,000 players simultaneously — without knowing if the game even works at that level. Report-Only mode is like turning on "Preview mode" — the policy evaluates every sign-in and RECORDS what it would have done (block/allow/MFA), but doesn't actually enforce anything. You watch the results for 2 weeks, see who would get blocked, fix exceptions, then switch to "Enforce." Zero production impact during testing.
  • 1
    Create the CA policy — configure all conditions (who, what app, what location, what action)
  • 2
    Set to "Report-Only" — policy evaluates but does NOT block anyone
  • 3
    Monitor in Entra Sign-in logs for 2 weeks — filter by the policy name, see "Would have been blocked"
  • 4
    Add exclusions for legitimate users who would be incorrectly blocked (service accounts, break-glass accounts)
  • 5
    Switch to "Enabled" — now the policy enforces. Confident, tested, no surprises.
🎯 Interview Question — Frequently Asked
Q: "A new Conditional Access policy was deployed and now 200 users can't access Teams. What do you do?"
Answer that shows experience: "First thing — don't panic, but act fast because this is a P1/P2 incident. My immediate action is to go to Entra → Security → Conditional Access → find the newly deployed policy → switch it to Report-Only mode (NOT disable it, because we still want the logs). This stops the blocking immediately while preserving the investigation data.

While users are regaining access, I run the What If tool in Conditional Access — I simulate a sign-in with a sample affected user to see exactly which policy is blocking them and why. Common causes: the policy was scoped too broadly (All Users instead of a specific group), a Named Location wasn't configured correctly so office IPs are being treated as untrusted, or service accounts were caught in the policy scope.

Once identified, I fix the policy configuration — add proper exclusions, fix Named Location, narrow the user scope — then move it from Report-Only back to Enabled. Throughout, I'm creating an incident ticket documenting the impact, root cause, and resolution timeline. This is standard ITSM practice for a P1 change-induced incident."
4
Break Glass Accounts — The Emergency Exit

Every organization must have 2 Break Glass accounts — emergency Global Admin accounts that are excluded from ALL Conditional Access policies. If every CA policy is wrong and locks everyone out, these accounts are the only way back in.

🔥 Fire Extinguisher Analogy
Fire extinguishers in buildings are locked in glass boxes — you need to break the glass to access them. This inconvenience is intentional — you only break it in a genuine emergency. Break Glass accounts work the same way: stored in a physical safe, very long complex password, no MFA requirement (because MFA might also be broken in the emergency), and every sign-in sends an alert to all admins immediately. These accounts exist ONLY for genuine lockout emergencies.

Identity & Security Questions
🎯 Q1 — Very Commonly Asked
Q: "Explain how SSO works in Microsoft 365. What happens when a user signs in?"
"When a user opens portal.office.com, their browser is redirected to login.microsoftonline.com — Microsoft's Entra ID authentication endpoint. The user enters their username and password. Entra ID validates the credentials against its directory (or against on-premises AD via AAD Connect in a hybrid setup). If MFA is enabled — and it should always be — a second factor challenge is sent: typically a push notification to the Microsoft Authenticator app.

Once all factors are verified, Entra ID issues an Access Token — a digitally signed JSON web token (JWT) with claims about the user's identity and permissions. This token is passed to each Microsoft 365 application the user opens. Applications like Teams, Outlook, and SharePoint validate the token without calling Entra ID again for each request. This is Single Sign-On — sign in ONCE, access everything. The token lasts 1 hour; a Refresh Token silently renews it in the background for up to 90 days without requiring re-login."
🎯 Q2 — Tests Real Knowledge
Q: "What is Conditional Access? Give me a real example of when you'd use it."
"Conditional Access is Microsoft's policy engine that evaluates every sign-in attempt using contextual signals and then decides whether to grant access, grant with conditions, or block. I describe it as a smart security guard who doesn't just check your badge — he looks at who you are, where you're coming from, what device you're using, and what you're trying to access.

Real example: A financial services company I studied has this CA policy: 'For Exchange Online access, if the user is not connecting from a corporate IP address AND not using a compliant corporate device, require MFA.' This means: in the office on a company laptop — no MFA prompt (trusted location + compliant device). From home on a personal laptop — MFA required. From an unknown country — blocked entirely.

Critical lesson from my preparation: ALWAYS deploy new CA policies in Report-Only mode first — it evaluates every sign-in and records what it WOULD have done, without actually blocking anyone. I had studied the scenario where a CA policy was deployed directly to production and locked out 500 users. Report-Only mode for 2 weeks before enabling is non-negotiable."

Exchange & Mail Flow Questions
🎯 Q3 — Tests Deep Technical Knowledge
Q: "A user reports that their emails to external recipients are going to spam. Walk me through your troubleshooting."
"I approach this systematically. First, I ask the user to get the full email headers from the recipient's spam folder — in Outlook: File → Properties → Internet Headers. I paste these into Microsoft's Message Header Analyzer at aka.ms/mha.

In the analyzer, I check three things in order: Authentication-Results shows me if SPF, DKIM, and DMARC passed or failed. If spf=fail, the sending server's IP isn't in the SPF record. If dkim=fail, DKIM isn't enabled for this domain in Exchange Admin Center. If dmarc=fail, it usually follows from SPF or DKIM failing.

For a new M365 migration, the most common root cause is DKIM not being enabled post-migration. The fix: Exchange Admin Center → Email Authentication → DKIM → enable for the domain → add the two CNAME records to DNS. Allow 24–48 hours for propagation.

I also check MXToolbox to see if the organization's sending IP is on any real-time blacklists — this happens when the organization previously had a mail server that sent spam, and the IP's reputation is still damaged.

Long-term fix: configure DMARC with p=quarantine, monitor DMARC reports for 4 weeks, then move to p=reject once all legitimate senders are covered."
🎯 Q4 — Tests Hybrid Knowledge
Q: "What is Exchange Hybrid and what is the Hybrid Configuration Wizard?"
"Exchange Hybrid is a coexistence state where some mailboxes are in on-premises Exchange Server and some are in Exchange Online, but they appear seamlessly integrated to users — they can see each other's free/busy calendar information, move mailboxes between on-prem and cloud, and share the same Global Address List.

The Hybrid Configuration Wizard (HCW) is a Microsoft-provided tool that automates setting up this coexistence. It configures: OAuth authentication trust between on-prem Exchange and EXO (so they trust each other's tokens for free/busy), inbound and outbound mail connectors for mail routing in hybrid, and Organization Relationships that enable free/busy sharing.

SharePoint, Teams & OneDrive Questions
🎯 Q5 — Tests SPO Depth
Q: "A SharePoint library has 6,000 files and users are getting an error when opening views. What's happening?"
"This is the classic SharePoint Online List View Threshold issue. SharePoint enforces a hard limit of 5,000 items that any single view query can return. When a library exceeds 5,000 items and a view doesn't have an indexed column filter, SharePoint's database engine refuses to run the query — it would have to scan the entire table, which would impact performance for all users on the shared platform.

I explain it with a library analogy: if you ask a librarian to show you ALL 50,000 books, they have to go through every shelf. If you ask for 'all Science books from the Science section,' they only look in one specific area. SharePoint needs that specificity for large libraries.

The fixes: First, add indexes to commonly filtered columns (Modified, Author, Content Type) in Library Settings → Indexed Columns. Second, update all views to filter using these indexed columns — for example, 'Items modified in last 30 days' using an indexed Modified column. Third, use Modern SharePoint views rather than Classic — they handle large libraries significantly better. For libraries over 100,000 items, restructure using folders, which effectively partition items into smaller groups that don't count against each other's threshold."
🎯 Q6 — Teams Infrastructure Knowledge
Q: "Why does Teams audio work perfectly at home but break at the office?"
"This is one of the most satisfying troubleshooting problems to diagnose because the root cause is almost always the same: the corporate network is blocking Teams real-time media traffic.

Teams uses UDP ports 3478-3481 and 50000-59999 for audio and video. UDP is fast — a few lost packets cause minor glitches but the stream continues. At home, your router lets Teams use UDP freely and it connects directly to Microsoft's nearest media relay servers.

In a corporate environment, the traffic often goes through a web proxy that forces everything through TCP (which inspects traffic but adds significant latency) or gets inspected by a firewall that times out UDP sessions. The result: choppy audio, video freezes, dropped calls.

Microsoft categorizes M365 traffic into three groups — Optimize, Allow, and Default. Teams real-time media is in the 'Optimize' category and MUST bypass the proxy entirely. The fix is configuring the proxy bypass list or VPN split-tunneling to exclude Microsoft's Optimize-category IP ranges, letting Teams media take the direct UDP path. I'd use the Teams Network Assessment Tool to verify packet loss and jitter before and after the fix — it connects directly to Microsoft's media relays and measures quality metrics."

Compliance & Security Questions
🎯 Q7 — Shows Compliance Knowledge
Q: "What is the difference between a Retention Policy and a Litigation Hold?"
"Great distinction — they're often confused but serve completely different purposes. I explain it with an analogy: a Retention Policy is like a government regulation that says all financial records must be kept for 7 years — it's a proactive, planned compliance requirement that applies broadly. A Litigation Hold is like a court order served on a specific person — it says 'you may not destroy any documents related to this legal matter, even if you normally would.'

Technically: a Retention Policy is configured in Microsoft Purview and can be applied to all mailboxes, all SharePoint sites, all OneDrive across the organization. Users can still delete items — but a hidden copy is preserved in the Recoverable Items\Preservation Holds folder that only compliance admins can access via eDiscovery.

A Litigation Hold is applied directly to a specific user's mailbox via Set-Mailbox -LitigationHoldEnabled $true. Once enabled, NOTHING the user deletes is permanently purged — everything moves to Recoverable Items\Purges and stays there indefinitely. It's a more targeted, stronger protection for active legal proceedings.

Key difference for the interviewer: Retention Policies are scheduled compliance planning. Litigation Hold is reactive — applied when legal action starts. Both should be in your toolkit, but for different purposes."
🎯 Q8 — The Scenario That Impresses Most
Q: "Walk me through how you'd handle a security incident where a user's account is compromised."
"When I detect or receive a report of a potentially compromised account, I follow a structured response — immediate containment first, investigation second, remediation third.

Immediate Containment (first 5 minutes): Run Revoke-MgUserSignInSession -UserId for the compromised account — this immediately invalidates all active tokens and forces re-authentication on every device and app. Reset the password to a complex temporary value. Enable MFA if it wasn't already on.

Investigation (next 30 minutes): Check Entra ID Sign-in logs for impossible travel events (signed in from India and Ukraine within 1 hour — impossible physically). Check the Unified Audit Log for: inbox rules created (common attacker technique to forward all email to external address), OAuth app consents (attacker may have consented a malicious app), mail sent from the account, SharePoint files accessed or downloaded.

Confirm the scope: Was this account used to access other accounts or systems? Check for lateral movement in security alerts.

Remediation: Remove any suspicious inbox forwarding rules (Get-InboxRule to check). Revoke any suspicious OAuth app consents in Entra → Enterprise Apps. Force re-registration of MFA — clear old methods, issue TAP for secure re-enrollment.

Prevent recurrence: Enable Entra ID Protection User Risk policy — set to require password change when User Risk = High. Enable Number Matching in Authenticator to prevent future MFA fatigue attacks.

Document everything with timestamps for the security incident report."
🎯 Q9 — L1/L2 Practical Question
Q: "A user calls saying 'I deleted an important file from SharePoint 2 months ago. Can you get it back?'"
"This is a recovery scenario with a critical timeline question. SharePoint Online's Recycle Bin retains deleted items for 93 days total — split between Stage 1 (the user's personal recycle bin) and Stage 2 (the Site Collection admin recycle bin). At 2 months (approximately 60 days), there's still a window of about 33 days remaining before permanent deletion.

My first action: Go to the SharePoint site → Site Settings → Recycle Bin → Check Stage 1 for the file. If the user already emptied their recycle bin, check Stage 2 — the site collection recycle bin. As a SharePoint admin, I navigate to the site → Site Settings → Recycle Bin → 'Second-stage recycle bin' link at the bottom. The file should be there.

If it's past 93 days: I would honestly tell the user it's likely permanently deleted from SharePoint's systems. I'd then check if there's a backup — does the organization use SharePoint Online backup solutions? Does Microsoft 365 Backup (Microsoft's native backup product) cover this site? Are there any export backups?

If there's truly no recovery path, I document the incident and recommend enabling a retention policy on the site so that future deletions are preserved for the full regulatory retention period — even if users delete from the recycle bin, a copy exists in the Preservation Hold Library."
